You've learned to classify streams, build composite curves, run the problem table algorithm, apply the golden rules, match and split streams into a feasible network, target the minimum number of units, and choose ΔTmin economically. This capstone puts you in the role of the process integration engineer, targeting new designs and auditing existing plants against the numbers this method gives you.

The thread through every case: **find the true minimum utility target before designing anything, and never let heat cross the pinch.** Most of the value in this discipline comes from two things, calculating a number nobody had calculated before, and spotting a violation of a rule that's easy to state but easy to miss on a real, messy, already-built site.

## How the cases work

Each is the same five-step loop as your other deep-dive capstones:

1. **Investigate**: read the brief and the panel: stream data, targets, an existing network, utility bills.
2. **Calculate**: work out the figure that decides it: a heat load, a minimum utility, a feasibility check, a unit count, a total cost. The **reference data is on hand**: the problem table method, the golden rules, the CP feasibility rule, minimum units, ΔTmin economics, but the *method* is yours. **"I'm stuck"** gives the formula then a nudge, but using it blocks a clean pass until you retry unaided.
3. **Diagnose**: pick the root cause from the evidence and your numbers.
4. **Prescribe**: build a work plan (reroute/reject infeasible matches first, then splits, then capital).
5. **Verify**: score across Calculation, Diagnosis, Prescription, Sequence and Capture, then read the debrief.

<Callout variant="key" title="The same signature, twice">
Notice cases 2 and 8 share the same underlying fault, a connection crossing the pinch, but arrive at it from opposite directions: one hands you the violation directly, the other hands you only utility bills and asks you to recognise the equal-gap signature that reveals it. Real retrofit auditing is almost always the second kind: nobody tells you what's wrong, only what the meters show.
</Callout>

## What you'll practise

- **Targeting**: the problem table algorithm on fresh stream data, and reading the balance.
- **The golden rules**: quantifying the exact utility penalty of a cross-pinch violation.
- **Feasibility & splitting**: the CP rule that decides which matches can be built, and when a stream must be split.
- **Threshold problems**: recognising when a process only ever needed one utility.
- **Economics**: comparing ΔTmin options on total annual cost, not utility cost alone.
- **Network simplification**: spotting a network with more units than its target, and retrofit auditing against a calculated gap.

## Differential thinking

A plant using "too much energy" might be untargeted from the start, carrying a golden-rule violation nobody's found, built around an infeasible match nobody checked, over-complicated with redundant units, or simply never audited against its own true minimum. Each has a different fix (a calculation, a re-route, a split, a loop-break) and the decisive evidence is always the same: does the number you calculate match what's actually installed?

## After you finish

- Which **number** decided each case: the cascade minimum, the CP comparison, the unit count, the utility gap?
- Could you reproduce each **calculation** on a real stream table, without the hints?
- For each fix, is it a near-free re-route, a design change (split, rebalance), or new capital (an exchanger), and would you recommend it in that order?
